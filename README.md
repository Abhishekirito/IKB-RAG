# IKB-RAG: Industrial Troubleshooting Assistant

A full-stack, React-based SaaS application integrated with a pure Python RAG (Retrieval-Augmented Generation) pipeline. This application utilizes **MinerU** for advanced document parsing (extracting complex tables, text, and layout), **PyMuPDF** for diagram extraction, and **Groq (Llama 3)** for high-speed industrial troubleshooting reasoning.

---

## 🚀 Setup & Installation Instructions

To run this application locally, you need to start both the Python backend and the React frontend.

### 1. Prerequisites
- Python 3.9+
- Node.js & npm
- Valid API Keys for Hugging Face and Groq

### 2. Backend Setup (FastAPI & RAG Engine)
Open a terminal in the root folder (`PIKE-RAG-main`) and run:
```bash
# Install required Python dependencies
pip install fastapi uvicorn python-multipart requests pymupdf groq

# Set your environment variables (in your terminal or .env file)
# Windows PowerShell:
$env:HF_TOKEN="your_huggingface_token"
$env:GROQ_API_KEY="your_groq_api_key"

# Start the Backend Server
python api.py
```
*The backend will run on `http://localhost:8002`.*

### 3. Frontend Setup (React/Vite)
Open a **new** terminal window, navigate to the `frontend` folder, and run:
```bash
cd frontend

# Install Node modules
npm install

# Start the Vite Development Server
npm run dev
```
*The frontend will run on `http://localhost:5173`. Open this URL in your browser.*

## ⚙️ Core Architecture & Working Functions

This project is divided into a robust FastAPI backend (acting as the RAG engine) and a dynamic React frontend. Below is a detailed breakdown of every working function.

### A. Backend API Endpoints (`api.py`)
- **`POST /upload`**: Handles PDF uploads. Prefixes the file with the `chat_id` for isolation, passes it to the RAG engine for parsing and indexing, and safely cleans up temporary files.
- **`GET /documents`**: Returns a list of all uploaded PDFs associated with a specific `chat_id`.
- **`GET /documents/{chat_id}/{filename}`**: Serves the raw PDF file directly to the frontend for the built-in document viewer.
- **`GET /images/{chat_id}/{pdf_base}/{filename}`**: Dynamically serves diagrams and images extracted by PyMuPDF using a strict nested folder structure.
- **`POST /chat`**: Receives a user's question, queries the Vector DB for semantic context, and streams/returns the response generated by the Groq LLM.
- **`POST /chat/{chat_id}/refresh`**: Triggers a complete re-parse and re-index of all documents associated with that chat.
- **`DELETE /chat/{chat_id}`**: Acts as a garbage collector, permanently wiping all PDFs, extracted images, JSON data, and Vector DB embeddings linked to the `chat_id`.
- **`POST /clear`**: A global nuclear option that entirely wipes the `simple_vector_db.json` and deletes all storage directories.

### B. The RAG Engine (`pikerag/utils/rag_utils.py`)
- **`index_document(file_path, chat_id)`**: 
  - **MinerU Integration**: Sends the document to the MinerU API to extract perfect Markdown and JSON data.
  - **Local Storage**: Saves the raw JSON and Markdown inside `parsed_data/{chat_id}/{pdf_name}/`.
  - **PyMuPDF Extraction**: Rips out raw images/diagrams from the PDF and saves them to `static_images/{chat_id}/{pdf_name}/`.
  - **Smart Chunking**: Intelligently chunks the Markdown by paragraphs (`\n\n`). If a paragraph (like a massive table) exceeds 1,500 characters, it recursively splits it down by single newlines to prevent Groq API rate limits.
  - **Embedding**: Sends the chunks to a Hugging Face embedding space and saves the vectors locally to `simple_vector_db.json`.
- **`query(question, chat_id)`**: Embeds the user's question, performs a cosine-similarity search against the local vector database, and generates a context-aware prompt for the Groq LLM.
- **`delete_chat_data(chat_id)`**: Scans the `uploaded_documents`, `static_images`, and `parsed_data` directories, utilizing `shutil.rmtree` to eradicate any nested folders belonging to the specified chat.

### C. Frontend Features (`frontend/src/App.jsx`)
- **Persistent Sessions**: Uses `localStorage` (`pike_rag_chats`) to remember all chat histories, titles, and active contexts even after you refresh the browser.
- **Dynamic Processing Feed**: When a file is uploaded, an interactive 2x4 grid UI spawns in the chat feed simulating the 8-step MinerU processing pipeline (with glowing yellow/green status indicators).
- **System Message UI**: Renders server statuses, upload successes, and errors as sleek glassmorphic bubbles. Includes a functional `X` close button to dismiss them from the feed once finished.
- **Chat Management Hub**: 
  - **Auto-Rename**: Renames "New Chat" automatically based on the first uploaded file or the first user message.
  - **Delete Chat**: Features a dark-mode confirmation modal to prevent accidental deletion of documents.
  - **Refresh Chat**: Allows users to manually force a re-index of their files, complete with UI success/error handling.
- **Markdown & Visual Rendering**: Processes the LLM's response using `ReactMarkdown`, seamlessly rendering text, tables, and the dynamically linked extracted diagrams (`![Diagram](...)`).

## ☁️ Hugging Face Model Deployment & API Access

To keep the local environment lightweight (zero heavy C++ or GPU dependencies), this project relies on Hugging Face Spaces for heavy lifting. 

### 1. MinerU Parsing API
We deployed MinerU as a remote microservice on Hugging Face.
- **Deployment Method**: A Docker-based Hugging Face Space running the official MinerU extraction pipeline.
- **API Access**: The FastAPI backend communicates with the space via standard HTTP `POST` requests.
  - **Endpoint**: `https://[your-space-name].hf.space/file_parse` (or equivalent Gradio `/api/predict` routes).
  - **Auth**: Authenticated automatically using the `HF_TOKEN` passed in the request headers.

### 2. BGE Embeddings API
Instead of loading massive sentence-transformers locally, the embedding generation is also outsourced.
- **Deployment Method**: A lightweight Hugging Face Space running `BAAI/bge-large-en-v1.5` (or similar) wrapped in a FastAPI/Gradio server.
- **API Access**: 
  - **Endpoint**: `https://[your-space-name].hf.space/embed`
  - **Usage**: During both document indexing and user querying, our local backend sends text chunks to this remote URL and receives purely the high-dimensional float arrays (vectors) in return.

### How to Deploy Your Own:
1. Create a new **Space** on [Hugging Face](https://huggingface.co/spaces).
2. Choose **Docker** as the environment.
3. Upload the model server files (e.g., standard FastAPI wrapper for sentence-transformers).
4. Update the endpoint URLs located at the top of `pikerag/utils/rag_utils.py` to point to your newly created Space URLs.
